Part 1

Platform Overview

The BitcoinEkasi TSK Bitcoin Rewards Platform is two cooperating systems that together form a complete Bitcoin-powered incentive programme for a youth surf/skate/fitness club in Mossel Bay, South Africa.

1.3 Deployment Architecture

Both systems run on a single VPS as Docker containers orchestrated by docker compose, sitting behind an nginx reverse proxy that handles SSL termination (Let’s Encrypt) and routes traffic by domain name.

CI/CD is handled by GitHub Actions: on push to main, each repository builds a Docker image and pushes it to the GitHub Container Registry (GHCR). Deployment runs docker compose pull && docker compose up -d on the VPS to roll out the new image with zero manual steps.

2.2 Participant Management

Registration

An Administrator enters the participant’s SA ID number, names, and other details. The system parses the 13-digit SA ID using the Luhn checksum algorithm to:

  • Validate the number is genuine
  • Extract the date of birth (first 6 digits: YYMMDD)
  • Extract the gender (digit 7: 0–4 = Female, 5–9 = Male)

The TSK ID (TSK00001, TSK00002, …) is auto-generated sequentially inside a database transaction to prevent duplicates under concurrent writes.

Profile Fields

Participant profiles are divided into logical sections: Personal Details, School & Education, Guardian & Contact, Body Measurements, Documents, TSK Programme, and Payment. Administrators edit via a full-page form; Marshalls can only raise change requests.

Body Measurements

Weight, height, t-shirt size, shoe size, and wetsuit size each have an independent *UpdatedAt timestamp. The timestamp is set to now only when that specific field’s value changes — if weight is edited but height is not, only weightUpdatedAt advances. This lets staff see exactly when each measurement was last taken.

Status Lifecycle

A participant starts ACTIVE. When retired, the Administrator selects a reason from a standardised list:

  • At age 18
  • Relocated
  • Poor attendance
  • Lack of interest
  • Negative attitude or behaviour
  • Other (requires free-text explanation)

The retiredAt date is set to the current timestamp. On retirement, the monthly report for the current month is automatically regenerated to reflect the change (see Section 2.4).

Junior Coach System

Selected participants are designated Junior Coaches at Level 1, 2, or 3. This flag directly multiplies their monthly reward:

LevelMultiplierEffect on base 10,000 sat reward
15×50,000 sats
27.5×75,000 sats
310×100,000 sats

TSK Level Progression

The programme has named progression levels (e.g. Grom → Gremmie → Shredder …). Administrators advance participants through these levels on the edit form. Every change is recorded in TskLevelHistory as an immutable audit entry — the history is visible on the participant detail page as a timeline.

Change Requests

When a Marshall notices a profile error (e.g. wrong phone number), they submit a ParticipantChangeRequest. They specify the field, the current value, and the proposed correction. Administrators see a queue of pending requests and can approve (triggering the update) or reject with a note. This workflow prevents Marshalls from having direct edit access while still allowing them to surface data issues.

2.3 Attendance System

Session Creation

An Event (session) represents a single training day. Only one session can exist per SAST calendar day — the system enforces this with a unique index on the date field.

Available categories allow tracking what type of activity was held:

CategoryDisplay Label
SURFINGSurfing
FITNESSFitness
SKATINGSkating
BEACH_CLEAN_UPBeach Clean Up

Role-Based Flow

Marshalls see a mobile-optimised single-page view. On login, they are immediately redirected to today’s session (if one exists) or to a stripped-down creation form. The mobile header has a single sign-out button — no sidebar, no navigation.

Administrators see the full attendance overview: all historical sessions grouped by month, a session creation form on the right, and a link to today’s session if it already exists.

Marking Attendance

Inside a session, every ACTIVE participant is listed. Staff tap to toggle each participant between:

  • Present — counted towards rewards
  • Absent — default state, not counted
  • On Tour — away at a competition; recorded separately, treated as absent for percentage purposes

Changes are saved immediately on tap (optimistic UI with server sync).

Sessions UI — Monthly Grouping

The attendance overview groups sessions by calendar month with an expand/collapse chevron. The most recent month is open by default. Each month header shows the total number of sessions and the combined attendee count for that month. Collapsed months show only the header row.

Each session row shows: date, category badge (colour-coded), attendee count (present participants only), optional note, and action links.

Session Deletion

Administrators can delete a session provided its month has not been approved. If the month’s report is in APPROVED status, deletion is blocked with an error message. When a session is deleted:

  1. All AttendanceRecord rows for that session are deleted.
  2. The Event row is deleted.
  3. If other sessions remain in that month, the monthly report is automatically regenerated.
  4. If the deleted session was the only one in the month, the report is reset to PENDING with no entries.

2.4 Monthly Report & Reward Logic

This is the core business logic of the platform. Each month produces a report that says: "for each participant, here is how many sessions they attended, their attendance percentage, and how many sats they earned."

Report Generation — upsertMonthlyReport()

A single idempotent function (src/lib/upsert-report.ts) handles all report creation and updates. It is called automatically when:

  • A new session (Event) is created
  • A session is deleted
  • A participant is marked RETIRED

The function runs inside a database transaction and fully replaces all MonthlyReportEntry rows for the target month — making it safe to call repeatedly with no side effects.

Participant Inclusion Rules

Which participants appear in a given month’s report is determined by these rules:

Participant StateIncluded?Reasoning
ACTIVE, registered before month end Yes Normal active participant.
ACTIVE, registered after month end No Not yet a participant during the month.
RETIRED, retiredAt ≥ monthStart Yes — for retirement month only Was still participating at some point during the month; sessions up to retirement date count.
RETIRED, retiredAt < monthStart No Retired before this month began; fully excluded.
New-joiner rule: A participant who joins mid-month is included in the report. Their attendance for sessions before their registration date is naturally zero (no records exist). They are divided by the full month session count — they do not get a prorated denominator. This is intentional: the same rule applies to retired participants.

Attendance Calculation

// For each participant:
const attendableEvents = participant.retiredAt
  ? events.filter(e => e.date <= participant.retiredAt)
  : events;  // all month events for active participants

const totalEvents = events.length;   // ALWAYS the full month count

const attended = attendableEvents.filter(e =>
  attendedSet.get(participant.id)?.has(e.id)
).length;

const percentage = (attended / totalEvents) * 100;

The key design decisions:

  • attended only counts sessions up to the retirement date (retired participants cannot attend sessions after they left).
  • totalEvents is always the full count of sessions in the month — no participant gets a smaller denominator regardless of when they joined or when they retired. This creates a fair, consistent metric.

Report Lifecycle

  1. Generate: Report created or refreshed automatically on data changes. Status starts as PENDING. If the report was APPROVED and is regenerated, it resets to PENDING (approval revoked).
  2. Review: Administrator views the report — sees each participant, their attendance, percentage, and reward amount.
  3. Approve: Administrator approves the report. Blocked for the current or future months (only past months may be approved). On approval, the system creates a batch payout via the Bolt API and stores a Lightning invoice for the total.
  4. Pay: The payout transitions through statuses: unpaid → invoiced → paid as the Lightning payment is settled.

2.5 Payout Integration

Payment Methods

Each participant has a paymentMethod setting:

  • BOLT_CARD — reward sats are credited as an internal balance on the BoltCard Server. The participant spends using their NFC card at point-of-sale.
  • LIGHTNING_ADDRESS — reward sats are sent outbound via Lightning Network to the participant’s Lightning address (e.g. name@wallet.satoshi.place).

Monthly Report Payout Flow

When an Administrator approves a monthly report, TSK calls the Bolt Server’s POST /api/v1/payout/batch endpoint with a list of { userId, amountSats, payoutType } items. Bolt processes the batch:

  • For internal items: credits the user’s balance ledger directly.
  • For ln_address items: sends a real Lightning payment to the address and records the result.

Academic Payouts

Term-based school grade rewards separate from monthly attendance. An Administrator uploads school reports, enters each participant’s academic percentage, and the system calculates a reward based on grade thresholds. These flow through the same Bolt batch payout mechanism.

Special Payouts

Ad-hoc reward batches for specific events (competition prizes, community participation, etc.). An Administrator creates a SpecialPayout with a title and description, then adds entries for individual participants with custom amounts.

2.6 Time Zone Strategy

South Africa uses SAST (UTC+2) with no daylight-saving time. The platform stores all dates in UTC but must correctly identify which SAST calendar day each event belongs to.

Convention: All Event dates are stored as noon UTC on the SAST calendar date. For example, "Monday 7 April 2026 SAST" is stored as 2026-04-07T10:00:00.000Z. This ensures the date never crosses midnight UTC in either direction when interpreted in SAST.

Month boundary calculations in src/lib/sast.ts:

  • getStartOfSASTMonth("2026-04") returns 2026-03-31T22:00:00.000Z (midnight SAST = UTC-2h)
  • getEndOfSASTMonth("2026-04") returns 2026-04-30T21:59:59.999Z (end of last SAST day)

All date-range queries in Prisma use these UTC boundary values, ensuring the database queries correctly capture the full SAST month regardless of when the query is executed.

BoltCard Server — bolt.bitcoinekasi.xyz

The BoltCard Server is an Express.js application that manages a Bitcoin balance ledger, handles NFC card payments, accepts Lightning Network refills, and exposes an admin dashboard and a payout API.

Eleven SQLite tables (created synchronously with better-sqlite3 on startup):

Card Keys Explained

Each BoltCard has 5 AES-128 (128-bit) keys written during programming:

KeyRole
K0Card authentication master key (used by card internally)
K1Session key — used by the server to decrypt the encrypted UID and tap counter from the NFC URL
K2CMAC key — used to verify the message authentication code in the NFC URL, preventing replay attacks
K3Reserved / factory use
K4Reserved / factory use

3.2 Card Programming Workflow

Before a participant can use their BoltCard, it must be programmed with the server’s keys and URL.

  1. Administrator opens the user detail page in the Bolt admin dashboard and clicks "Generate Card QR".
  2. The server generates 5 random 128-bit AES keys (K0–K4) and stores them against the user’s card record. The card UID is initially unknown.
  3. The server constructs a configuration payload encoding the keys and the LNURL-W endpoint URL, and returns it as a QR code.
  4. An administrator scans the QR code with the BoltCard Programmer mobile app (available for iOS/Android).
  5. The app establishes an NFC connection with the blank card and writes all 5 keys plus the LNURL-W URL to the card’s NTAG424 memory.
  6. On the card’s first tap at a payment terminal, the server decrypts the NFC payload and captures the card’s real UID, completing registration.

To decommission a card, the administrator generates a "wipe QR" which resets the card to factory keys.

3.3 Card Tap → Payment (LNURL-W)

LNURL-Withdraw (LNURL-W) is the Lightning Network protocol for pulling a payment from a server to a wallet. When a participant taps their BoltCard at a point-of-sale terminal, the following happens:

  1. NFC tap: The card broadcasts a URL of the form https://bolt.bitcoinekasi.xyz/lnurlw?p=ENCRYPTED_PAYLOAD&c=CMAC.
  2. Server receives /lnurlw: The server extracts p (AES-encrypted UID + counter) and c (CMAC authentication tag).
  3. Decrypt: Using the card’s stored K1 key, the server decrypts p to recover the plaintext UID and 3-byte tap counter.
  4. Identify card: The server looks up the card by UID. On first tap, UID is unknown — the server identifies by the encrypted payload pattern and stores the UID.
  5. Verify CMAC: Using K2, the server recomputes the CMAC over the decrypted data and compares it to c. If they don’t match, the request is rejected as tampered.
  6. Counter check: The decrypted counter must be strictly greater than the last seen counter stored in the DB. This prevents replay attacks (tapping a recorded NFC signal a second time does nothing).
  7. Return LNURL-W response: Server responds with a JSON payload containing the withdrawal parameters: minWithdrawable, maxWithdrawable (the user’s balance up to the per-transaction limit), defaultDescription, and the callback URL.
  8. Wallet creates invoice: The payment terminal or wallet creates a Lightning invoice for the requested amount and sends it to /lnurlw/callback?k1=TOKEN&pr=INVOICE.
  9. Callback processing: Server decodes the invoice to confirm the amount, checks it doesn’t exceed the user’s balance or daily/per-tx limits, then calls Blink’s lnInvoicePaymentSend GraphQL mutation.
  10. Balance deduction: On successful payment, the user’s balance_sats is decremented and a transactions row is inserted.
Replay prevention: The strictly-increasing counter (stored on the card’s secure memory and verified server-side) means that even if someone records an NFC tap, re-broadcasting the same URL would fail the counter check. The CMAC further ensures the payload hasn’t been tampered with.

3.4 Balance Refill (LNURL-P)

Participants can top up their BoltCard balance by sending Lightning payments to their Lightning address (e.g. username@bolt.bitcoinekasi.xyz).

  1. Lightning address lookup: The sender’s wallet fetches https://bolt.bitcoinekasi.xyz/.well-known/lnurlp/username and receives a LNURL-P metadata response.
  2. Invoice request: The sender’s wallet requests a Lightning invoice for their chosen amount via /lnurlp/username/callback?amount=MILLISATS.
  3. Invoice creation: The server calls Blink’s lnInvoiceCreate GraphQL mutation to create an invoice on the Blink account, then stores the invoice in pending_refills (with 1-hour expiry) and returns it to the sender’s wallet.
  4. Payment: The sender pays the invoice. Blink detects the incoming payment.
  5. Real-time notification: The Blink WebSocket subscription fires a LnUpdate event containing the payment hash and settled amount.
  6. Balance credit: The server matches the payment hash to a pending_refills row, credits the corresponding user’s balance_sats, inserts a transactions row, and removes the pending_refill entry.

A background job runs every 30 minutes to clean up expired (unpaid) pending_refills.

3.5 Blink Lightning Integration

All Lightning Network operations are handled through Blink (galoy.io), a custodial Bitcoin Lightning wallet that exposes a GraphQL API and a WebSocket subscription feed.

GraphQL HTTP API

WebSocket Subscription

The server maintains a persistent WebSocket connection to Blink’s subscription endpoint, subscribing to myUpdates. When a payment arrives, Blink pushes a real-time LnUpdate event:

subscription {
  myUpdates {
    update {
      ... on LnUpdate {
        paymentHash
        status
        settlementAmount
      }
    }
  }
}

The server uses graphql-ws with retryAttempts: Infinity to automatically reconnect if the WebSocket drops.

Transaction Counterparty Extraction

The admin dashboard shows a "From/To" column on Blink transactions. The server extracts the counterparty from the settlementVia field:

  • IntraLedger (Blink-to-Blink): shows the counterparty’s Blink username
  • OnChain: shows the Bitcoin transaction hash
  • Lightning: shows "Lightning" (external LN payment; no identifying info available)

3.7 Batch Payout API

TSK calls Bolt’s POST /api/v1/payout/batch endpoint when a monthly (or other) payout is approved. This endpoint is protected by an API key (hashed with SHA-256 in the DB).

Request Format

POST /api/v1/payout/batch
Authorization: Bearer <api_key>

{
  "batchId": "tsk-2026-03",
  "description": "Monthly rewards — March 2026",
  "items": [
    { "userId": "TSK00001", "amountSats": 10000, "payoutType": "internal" },
    { "userId": "TSK00042", "amountSats": 8200,  "payoutType": "ln_address" }

}

Processing

  • internal items: balance is credited immediately to the user’s account. A payout_batch_items row is marked completed.
  • ln_address items: the server initiates a Lightning payment to the user’s stored Lightning address. The outbound payment is tracked in ln_payouts. On WebSocket confirmation from Blink, the item is marked completed.

Part 4

How the Two Systems Work Together

TSK Rewards and the BoltCard Server are loosely coupled via a simple REST API. TSK is the source of truth for participants and rewards; Bolt is the source of truth for balances and card payments.

End-to-End Reward Flow

  1. Attendance tracked (TSK): Throughout the month, Marshalls mark daily attendance. Each session auto-updates the monthly report entry for each participant.
  2. Month closes (TSK): At the end of the month, an Administrator opens the monthly report. The report shows every participant, their attendance %, and calculated reward in sats.
  3. Report approved (TSK → Bolt): Administrator clicks Approve. TSK calls POST /api/v1/payout/batch on the Bolt server, passing each participant’s TSK ID (mapped to their Bolt username) and reward amount.
  4. Balances credited (Bolt): Bolt processes the batch. For BOLT_CARD participants, their internal balance increases. For LIGHTNING_ADDRESS participants, Bolt sends an outbound Lightning payment.
  5. Spending (Bolt): Participants with BoltCards tap to pay at community vendors, sports shops, or programme-affiliated merchants. Each tap deducts from their internal balance via LNURL-W.
  6. Lightning refills (Bolt): Any Lightning-capable wallet can send sats to username@bolt.bitcoinekasi.xyz to top up a participant’s balance directly, without going through TSK.

Data Mapping

Independence

The systems can operate independently. The Bolt server has no knowledge of attendance or reward calculations — it just processes the batch it receives. TSK has no knowledge of card balances or tap history. This separation of concerns keeps each system simple and allows either to be replaced or upgraded independently.

5.3 Server Deployment

The VPS runs a single docker-compose.yml in the /home/user/infra/ directory that defines both services:

services:
  tsk:
    image: ghcr.io/org/tsk_rewards:latest
    restart: unless-stopped
    environment:
      - DATABASE_URL=file:/data/tsk.db
      - NEXTAUTH_SECRET=${NEXTAUTH_SECRET}
      - NEXTAUTH_URL=https://tsk.bitcoinekasi.xyz
    volumes:
      - tsk_data:/data
      - tsk_uploads:/app/public/uploads

  bolt:
    image: ghcr.io/org/bolt:latest
    restart: unless-stopped
    environment:
      - JWT_SECRET=${JWT_SECRET}
      - BLINK_API_KEY=${BLINK_API_KEY}
      - BLINK_WALLET_ID=${BLINK_WALLET_ID}
    volumes:
      - bolt_data:/data

To deploy a new version:

docker compose pull
docker compose up -d

Docker Compose compares the new image digest to the running container and only restarts services where the image changed.

5.4 nginx Configuration

Two server blocks — one per domain. Key configuration:

server {
    server_name tsk.bitcoinekasi.xyz;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

server {
    server_name bolt.bitcoinekasi.xyz;

    location /assets/ {
        proxy_pass http://localhost:5173;
        proxy_buffering off;   # Critical: prevents truncation of large JS bundles
    }

    location / {
        proxy_pass http://localhost:3001;
        proxy_set_header Host $host;
    }
}
proxy_buffering off on the /assets/ location is critical. Without it, nginx’s default 32KB proxy buffer truncates large JavaScript bundles, causing a blank white page. Setting proxy_buffering off streams the file directly to the browser without buffering.

SSL is managed by Certbot (Let’s Encrypt), which adds the ssl_certificate and ssl_certificate_key directives and sets up automatic HTTPS redirect.

5.6 Database Migrations

TSK (Prisma): The Docker entrypoint runs prisma migrate deploy before starting the Next.js server. This applies any pending SQL migrations from prisma/migrations/ to the production database safely — skipping already-applied migrations and never touching data.

Bolt (SQLite): On startup, the Express app runs synchronous CREATE TABLE IF NOT EXISTS statements for all 11 tables. This is idempotent — existing tables are untouched. Schema changes require a manual ALTER TABLE migration or a data migration script run on the server.

5.7 Maintenance Notes

  • Disk space: Docker image layers accumulate on the VPS over time. Run docker system prune -f before deploying if disk space is low.
  • SQLite backups: Both databases are single files on Docker named volumes. Back up by running docker cp <container>:/data/tsk.db ./backup-tsk.db.
  • Log access: docker compose logs -f tsk or docker compose logs -f bolt for live logs.
  • Seed users: After a fresh TSK deployment, run docker compose exec tsk npm run db:seed to create the default admin and marshall accounts.
